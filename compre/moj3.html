The Executioner's Stack

These newly uncovered questions from the archives test your understanding of stack unwinding, static persistence across recursive frames, and operating-system-level file I/O overrides.

🧠 Advanced Concept Reminders

  • Static Recursion: A static variable inside a recursive function does NOT reset. Its state persists across every layer of the call stack.
  • Pass-By-Value Illusion: If you pass an integer into a recursive function and it gets modified, the original variable in `main` remains completely untouched.
  • Append Mode Absolute: Using rewind() or fseek() on a file opened in `"a"` mode does nothing to `fprintf/fputs`. The OS forces ALL writes to the EOF.
  • Pointer Arguments: Modifying a pointer variable (like p++) inside a function doesn't affect the caller's pointer, but modifying what it points to (p[0] = 5) absolutely does.

I. The Recursion Labyrinth

Mind-Bender

1. The Static Phantom (2024-25)

Track the state of `r` very carefully across the stack frames.

#include <stdio.h>

int f(int n) { 
    static int r = 0; 
    if (n <= 0) return 1; 
    
    if (n > 3) { 
        r = n; 
        return f(n - 2) + 2; 
    } 
    return f(n - 1) + r; 
}

int main() {
    printf("%d", f(5));
    return 0;
}

Answer: 18

The Trick: The variable `r` is static. It is shared across ALL recursive calls. Let's trace it:

  • f(5): `n>3` is true. `r` becomes 5. Returns `f(3) + 2`.
  • f(3): `n>3` is false. Returns `f(2) + r`. Since `r` is static, it is STILL 5! Returns `f(2) + 5`.
  • f(2): `n>3` is false. Returns `f(1) + 5`.
  • f(1): `n>3` is false. Returns `f(0) + 5`.
  • f(0): Base case hit. Returns `1`.

Unwinding the stack:
f(1) = 1 + 5 = 6
f(2) = 6 + 5 = 11
f(3) = 11 + 5 = 16
f(5) = 16 + 2 = 18.

Illusion

2. The Stack Unwind Mirage (2016-17)

#include <stdio.h>

void foo(int n, int sum) { 
    int k = 0, j = 0; 
    if (n == 0) return; 
    
    k = n % 10; 
    j = n / 10; 
    sum = sum + k; 
    
    foo(j, sum); 
    printf("%d,", k); 
}

int main() {
    int a = 2048, sum = 0;
    foo(a, sum);
    printf("%d", sum);
    return 0;
}

Answer: 2,0,4,8,0

The Trick: Two separate traps here:

  1. Post-Recursion Print: The printf("%d,", k) happens after the recursive call. This means it prints while the stack is unwinding (in reverse order). It extracts digits 8, 4, 0, 2, but prints them as 2, 0, 4, 8,.
  2. Pass-By-Value: Look at `main()`. It passes `sum` to `foo()`. Even though `foo` accumulates the sum internally, it takes `sum` by value, not by pointer. The original `sum` in main is completely unaffected and remains `0`!

II. Pointer Aliasing & Mutation

Advanced

3. Multi-Dimensional Array Mutation (2022-23)

#include <stdio.h>

void modify(int (*b)[4]) {
    b[1][1] = 100;
    b++;
    b[0][1] = 10;
}

int main() {
    int a[2][4] = {{1, 2, 3, 4}, {5, 6, 7, 8}};
    modify(a);
    printf("%d %d\n", a[0][1], a[1][1]);
    return 0;
}

Answer: 2 10

The Trick: Understanding how pointer arithmetic shifts the "base" of an array view.

  • Initially, `b` points to `a[0]`.
  • b[1][1] = 100; alters `a[1][1]` (changes 6 to 100).
  • b++ increments `b`. Since `b` is a pointer to an array of 4 ints, it jumps forward by one whole row. `b` now points to `a[1]`.
  • b[0][1] = 10; modifies the 0th row relative to the new pointer. Since `b` points to `a[1]`, this modifies `a[1][1]` (changes 100 to 10).
  • `a[0][1]` was never touched. It remains 2.
Danger

4. The Out-of-Bounds Loop Skip (2022-23)

#include <stdio.h>

int main() {
    char a[7] = "PILANI"; 
    int m = 1; 
    
    for( ; a[m] != '\0'; m++) {
        if((m % 2) != 0) {
            printf("%c", a[++m]); 
        }
    }
    return 0;
}

Answer: L N (Followed by Undefined Behavior / Crash)

The Trick: The inline increment ++m inside the loop body completely bypasses the termination check.

  1. m=1. Odd. ++m makes m=2. Prints `a[2]` ('L'). Loop updates: m=3.
  2. m=3. Odd. ++m makes m=4. Prints `a[4]` ('N'). Loop updates: m=5.
  3. m=5. Odd. ++m makes m=6. Prints `a[6]` ('\0' - prints nothing). Loop updates: m=7.
  4. CRASH: The loop checks `a[7] != '\0'`. But `a` only has indices 0 through 6! Reading `a[7]` is a buffer overread (Undefined Behavior).

III. File I/O & OS Limitations

Expert

5. The Append Rewind Myth (2024-25)

What will be the final content of the file data.txt?

#include <stdio.h>

int main() {
    FILE *fp;
    
    fp = fopen("data.txt", "w");
    fputs("Hello\n", fp);
    rewind(fp);
    fputs("Bye", fp);
    fclose(fp);

    fp = fopen("data.txt", "a");
    fputs("Hello", fp);
    rewind(fp);
    fputs("Bye", fp);
    fclose(fp);
    
    return 0;
}

Answer (File Content): Byelo\nHelloBye

The Trick: Operating systems handle Write (`"w"`) and Append (`"a"`) modes fundamentally differently.

  • Write Mode block: Writes `Hello\n`. rewind(fp) moves the cursor back to 0. Writing `Bye` overwrites the first 3 letters. The file now contains `Byelo\n`.
  • Append Mode block: Appends `Hello` at the end -> `Byelo\nHello`. It then calls rewind(fp). HOWEVER, in Append mode, the C standard and OS POSIX standards dictate that all write operations are forced to the End-Of-File, regardless of where the file pointer is! So `Bye` is simply appended to the end.
Macro Logic

6. Nested Macro Arithmetic (2024-25)

#include <stdio.h>

#define fun(n) (n==0 ? 1 : (n * 2))
#define nuf(n) (n==0 ? 1 : (n - 2))

int main() {
    int n = 5;
    int result = fun(n * nuf(n) * fun(n));
    printf("Result = %d", result);
    return 0;
}

Answer: Result = 300

The Trick: Tracing the exact resolution of the nested arguments.

  • Evaluate nuf(5): (5==0 ? 1 : (5-2)) -> 3
  • Evaluate fun(5): (5==0 ? 1 : (5*2)) -> 10
  • The inner expression becomes: `5 * 3 * 10` = 150.
  • Evaluate the outer macro fun(150): (150==0 ? 1 : (150*2)) -> 300.
